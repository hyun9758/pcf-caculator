# 설계 결정 및 회고

발표용 논리적 설명 자료. 각 결정의 **이유**, **대안**, **trade-off**를 정리.

---

## 1. 데이터 레이어: Repository 패턴

### 결정
`lib/data/repository.ts`에서 `USE_DB` 환경변수로 In-memory / PostgreSQL 전환.

### 왜 이렇게 했나
- 2-3일 타임박스에서 **UI/UX와 도메인 로직에 집중**하기 위해 In-memory로 시작
- 동시에 "DB 전환이 쉬운 구조"를 보여주기 위해 Repository 추상화
- 실제로 Prisma + Postgres 연동까지 완료 → **두 모드 모두 동작**

### 대안
| 대안 | 장점 | 단점 |
|------|------|------|
| 처음부터 Postgres | 완전한 풀스택 | 세팅에 시간 소모, Docker 필요 |
| JSON 파일 | 영속성 있음 | 동시성 문제, 타입 안전성 떨어짐 |
| SQLite | 설치 불필요 | Prisma 호환성, 프로덕션 부적합 |

### Trade-off
In-memory는 서버 재시작 시 데이터 리셋. 하지만 Repository 패턴 덕분에 **코드 한 줄(USE_DB=true) 변경**으로 Postgres 전환 가능. 이 아키텍처 판단이 핵심.

## 2. 경영자 vs 실무자 뷰: Context 기반 토글

### 결정
`ViewModeContext`로 동일 페이지에서 조건 렌더링. 별도 URL/페이지 아님.

### 왜 이렇게 했나
- 두 사용자가 **같은 데이터를 다른 깊이**로 봄
- 경영자가 실무자 뷰로 전환해서 상세 확인하는 시나리오
- URL이 같으므로 링크 공유 시 혼란 없음

### 대안
| 대안 | 장점 | 단점 |
|------|------|------|
| 별도 페이지 (/exec, /prac) | URL로 뷰 구분 | 코드 중복, 같은 데이터 두 번 페칭 |
| 쿼리 파라미터 (?view=exec) | URL 공유 가능 | 뷰 전환마다 페이지 리로드 |
| Context 토글 (선택) | 즉시 전환, 코드 재사용 | URL에 상태 미반영 |

### Trade-off
URL에 뷰 상태가 없어서 북마크/공유 시 항상 기본(경영자) 뷰. 실제 서비스라면 쿼리 파라미터 방식이 나을 수 있지만, 대시보드 특성상 "빠른 전환"이 더 중요하다고 판단.

## 3. 차트 라이브러리: Nivo

### 결정
@nivo/sankey, @nivo/bar, @nivo/pie, @nivo/line 사용.

### 왜 이렇게 했나
- **Sankey 다이어그램이 핵심** — PCF 전과정 흐름 시각화에 최적
- Nivo는 Sankey를 네이티브 지원하는 유일한 React 차트 라이브러리
- D3 기반이지만 선언적 React API → 빠른 개발

### 대안
| 대안 | Sankey 지원 | React API | 개발 속도 |
|------|-----------|-----------|----------|
| Nivo (선택) | ✅ 네이티브 | ✅ 선언적 | 빠름 |
| Recharts | ❌ 없음 | ✅ 선언적 | 빠름 |
| D3.js | ✅ 직접 구현 | ❌ 명령형 | 느림 |
| Chart.js | ❌ 없음 | △ 래퍼 | 보통 |
| ECharts | ✅ 있음 | △ 래퍼 | 보통 |

### Trade-off
Nivo 번들 사이즈가 크지만 (4개 차트 패키지), Sankey + 일관된 API가 주는 개발 속도 이점이 더 큼. 프로덕션에서는 tree-shaking으로 최적화 가능.

## 4. Sankey 다이어그램 데이터 변환

### 결정
PCF lineItems → 3단계 Sankey: **입력 소스** → **전과정 단계** → **총 배출량**

### 왜 이렇게 했나
- 탄소 회계의 핵심 질문: **"어디서 얼마나 배출되나?"**
- Sankey는 **흐름의 크기**를 직관적으로 보여줌
- 비전문가도 "굵은 선 = 많은 배출"로 이해

### 구현 난점
- PcfLineItem → Sankey 노드/링크 변환이 복잡
- 같은 배출계수가 여러 활동에 사용될 수 있음 → 그룹핑 필요
- 음수 값(재활용 크레딧) 처리 → 양수만 Sankey에 포함

### 대안
Waterfall 차트가 더 구현 쉽지만, Sankey가 "흐름" 개념을 더 잘 전달. PCF는 본질적으로 투입→변환→배출 흐름이므로 Sankey가 도메인에 가장 적합.

## 5. Mock 데이터 설계: 현실성 vs 편의성

### 결정
ecoinvent, IPCC, 환경부 데이터 기반 사실적 수치. 5개 제품 × 28개 배출계수.

### 왜 이렇게 했나
- **도메인 이해도 25%** → 데이터 자체가 도메인 지식의 증거
- 각 제품이 다른 탄소 프로필 → 시각화 다양성 확보
  - 열연강판: Scope 1 지배 (고로 공정)
  - 노트북: Scope 3 지배 (글로벌 공급망)
  - 배터리셀: Scope 2 지배 (전력 집약 제조)
- 재활용 크레딧(음수) 포함 → 순환경제 이해 시연

### 데이터 검증
| 제품 | 우리 데이터 | 업계 참고값 | 근거 |
|------|-----------|-----------|------|
| 열연강판 1ton | ~1,855 kgCO₂e | 1,800-2,200 | worldsteel 2022 |
| 노트북 1대 | ~350 kgCO₂e | 300-400 | Dell/HP ESG 보고서 |
| PET병 1,000개 | ~85 kgCO₂e | 80-120 | PET 산업 LCA |

## 6. 감축 시뮬레이션: 시나리오 설계

### 결정
배출계수에 감축 비율 곱하기. 복수 시나리오 누적 곱.

### 왜 이렇게 했나
- 실무에서 감축 계획 수립 시 "이 시나리오 적용하면 얼마나 줄까?" 질문
- 단순 곱하기지만 **비즈니스 가치가 높은 기능**
- 복수 시나리오 조합 → 현실적 (재생에너지 + 운송 최적화 동시 적용)

### 4가지 시나리오 선정 근거
| 시나리오 | 근거 |
|----------|------|
| 재생에너지 전환 | RE100 가입 기업 증가, Scope 2 감축 핵심 |
| 재생 원소재 확대 | 순환경제 정책, EU CBAM 대응 |
| 운송 최적화 | 모달 시프트(트럭→철도), 거리 단축 |
| 에너지 효율 개선 | 가장 비용 효율적인 감축 수단 |

### Trade-off
실제로는 감축 비율이 선형이 아니고, 투자 비용/시간도 고려해야 함. 하지만 데모에서는 "방향성"을 보여주는 것이 목적이므로 단순화.

## 7. API 설계: REST vs 직접 Import

### 결정
API 라우트 10개 구현. 프론트에서는 일부 직접 import (클라이언트 컴포넌트 성능).

### 왜 이렇게 했나
- **시스템 설계 30%** → API 레이어 존재 자체가 설계 역량 증명
- 대시보드/제품 목록은 클라이언트에서 직접 계산 (SSG 가능)
- 상세/CRUD는 API 경유 → 서버 사이드 계산

### 현실적 타협
모든 페이지를 API fetch로 전환하면 더 "올바른" 아키텍처지만, 클라이언트 컴포넌트에서 직접 import가 초기 로딩 빠름. 프로덕션에서는 React Server Components + API 조합이 최적.

## 8. Prisma 7 + PostgreSQL

### 결정
Prisma 7 + PrismaPg adapter + Docker Compose.

### 왜 이렇게 했나
- 과제에 "Optional: Postgres" 명시 → 가산점
- Repository 패턴으로 **기존 코드 변경 없이** DB 연동
- Seed 스크립트로 mock 데이터 자동 투입

### 스키마 설계
```
Product 1:N ActivityData N:1 EmissionFactor
```
- ActivityData가 중심 테이블 (제품과 배출계수를 연결)
- Cascade Delete: 제품 삭제 시 활동 데이터도 삭제
- snake_case 테이블명 (PostgreSQL 관례)

## AI 활용 상세

### 사용 도구
- **Claude (Anthropic)** — 구현 계획, 코드 작성, 문서

### AI가 한 것
| 작업 | AI 기여도 | 설명 |
|------|----------|------|
| 프로젝트 구조 설계 | 높음 | 디렉토리 구조, 파일 분리 제안 |
| TypeScript 타입 | 높음 | 타입 정의 코드 생성 |
| 컴포넌트 코드 | 높음 | Nivo 차트, shadcn/ui 조합 |
| Mock 데이터 | 중간 | 구조 생성, 수치는 공개자료 참조 |
| README | 높음 | 초안 작성 |

### 내가 판단한 것
| 결정 | 설명 |
|------|------|
| 도메인 모델 | 어떤 타입이 필요한지, 관계 구조 |
| 뷰 분기 전략 | 경영자/실무자를 어떻게 분리할지 |
| Sankey 채택 | 왜 Sankey가 PCF에 적합한지 |
| 배출계수 검증 | 생성된 수치가 현실적인지 확인 |
| UX 흐름 | 어떤 정보를 어디에 배치할지 |
| 감축 시나리오 | 어떤 시나리오가 실무적으로 의미 있는지 |
| Repository 패턴 | In-memory와 DB를 어떻게 공존시킬지 |

### AI 활용에서 배운 점
- AI는 **코드 생성 속도**를 극대화하지만, **도메인 판단**은 사람이 해야 함
- 생성된 배출계수 값을 그대로 쓰면 안 됨 → 공개 자료와 교차 검증 필수
- AI가 제안한 구조를 맹목적으로 따르지 않고, 평가 기준에 맞게 조정

## 돌아보며

### 잘한 것
- **도메인 모델을 먼저 설계** → 타입이 가이드라인 역할, 이후 구현이 자연스러움
- **Sankey 다이어그램** → PCF에 가장 적합한 시각화, 차별화 포인트
- **Repository 패턴** → In-memory로 빠르게 시작, Postgres로 확장 가능 증명
- **감축 시뮬레이션** → 단순 시각화를 넘어 실무 가치 제공

### 아쉬운 것
- 프론트에서 직접 import와 API fetch가 혼재 → 일관성 부족
- 데이터 입력 CRUD가 클라이언트 상태만 → DB 연동까지 하면 더 완전
- 테스트 코드 미작성 → 계산 엔진 단위 테스트 있었으면 설계 점수 더 높았을 것
- 시계열 데이터가 하드코딩 → DB에서 집계하는 구조가 더 현실적

### 시간이 더 있었다면
- 계산 엔진 단위 테스트 (Jest)
- E2E 테스트 (Playwright)
- CSV/Excel 업로드 기능
- PDF 보고서 생성 (ISO 14067 형식)
- 실시간 협업 (WebSocket)
